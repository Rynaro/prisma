# Fixture conventions

This directory holds the per-scenario evaluation fixtures consumed by
`@prisma-bot/eval-runner`. The runner reads each `<id>.yaml` file, validates
it against `ScenarioFixtureSchema` (see `evals/runner/src/schema.ts`), and
runs the Phase 5 orchestrator (`runPipeline`) against it with a `FakeProvider`
and a hand-rolled `OctokitLike`.

## Layout

- `evals/scenarios.yaml` — the index. Lists each scenario's `id`, `name`,
  `fixture` path (relative to `evals/`), and `tags`.
- `evals/fixtures/<id>.yaml` — the per-scenario fixture. Carries
  `config_overrides`, `pr_payload`, `octokit_responses`, `provider_script`,
  `expectations`, and `metrics`.
- `evals/fixtures/<id>/` — optional auxiliary payloads. Created only when a
  fixture references files via `from_file`.

## Auxiliary payloads (`from_file`)

When an `octokit_responses` field would otherwise inline a large array, the
fixture YAML can use a one-key object `{ from_file: <relative-path> }`. The
runner resolves the path relative to `evals/fixtures/` and substitutes the
parsed JSON value before zod validation runs. Only `pulls_list_files` uses
this convention today.

The convention exists for `oversized-pr/files.json`, where 6 file entries
would otherwise dominate the YAML. Other scenarios inline their data.

## Naming

- The `id` field in the fixture YAML MUST equal the basename of the file
  without the `.yaml` extension (the loader asserts this on load).
- The `id` MUST equal the `id` in the matching `evals/scenarios.yaml` entry.
- Aux directories use the same `<id>` as the fixture they belong to.
